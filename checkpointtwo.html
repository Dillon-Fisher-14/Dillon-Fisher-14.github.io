Two-Dimensional Model


Since the last checkpoint, I have updated my gap filling algorithm to work with two-dimensional input. The benefit to this is that now every filled in pixel is influenced by every other pixel in the image. This will allow the resulting image to have more vertical continuity compared to the final image I showed in the previous checkpoint.

This version of the algorithm work in mostly the same way as the one-dimensional variant. Instead of only having one frequency parameter that I am varying by, I have two. And instead of fitting a sinewave the data I am fitting a product of sinewaves, of the form: y = a * sin(fx * px + cx) * sin(fy * py + cy). The variables fx and fy are the current frequencies being evaluated, px and py are the current pixel indices, and a, cx, and cy are the trainable parameters that the gradient descent is trying to calculate.

Robustness Tests


One of my design constraints is that the algorithm should work for arbitrarily filled in data. So, if enough data is present, it should be able to fill something realistic in. I devised two tests to see how my algorithm performs on differently filled in data. The first test gave each sample data point a percent chance of being removed from the initial data. The algorithm would run on the remaining points, as usual. The second test removed a percentage of the dataset from the middle of the data. This produced an increasing gap in the middle of the data, leaving only the two ends for high percentages.

My algorithm performed pretty well on the Uniformly Distributed Dropout, but the Center Gap Dropout has revealed a few flaws in how the algorithm behaves given certain inputs. Ideally, the algorithm should be able to take any input data and fill in the missing data. Moving Forward

Moving forward, I plan to make tweaks to the gap filling algorithm to make it more robust. The results from the extremely sparse data were worse than I had expected. Altimetry dataset with as little as 10% data is very rare (unless it is a landing site), but I would like this algorithm to one day be able to take a reasonably small dataset, like a landing site, and extrapolate a surface for an entire planet. Maybe it is a pipe dream, but I think it would be cool.

However, the algorithm works fine for the raw Venus dataset (which for reference has about 83% data). I am on track to complete the project by the end of the semester. After I make a few tweaks to the algorithm (hopefully for the better), I will run it on a larger image. The larger image will then be used to render the geometry of the planet.
